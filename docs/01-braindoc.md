# Braindoc — WanderFive

## Summary
WanderFive is a single-page web app: a user types any location (city, neighborhood, landmark, "near me" text), and the app returns exactly 5 curated places to visit there, each with a name and a short, useful blurb — rendered as polished, well-designed cards. Recommendations are generated by Claude itself at request time (no third-party places API, no API key setup), so the product is a pure LLM-reasoning app with a strong design layer as its main differentiator.

## The problem
Someone planning a trip, a weekend outing, or just curious about a place they're passing through wants a fast, opinionated shortlist — not a 40-tab research session across TripAdvisor, Google Maps, and Reddit threads. The current workaround (open several review sites, cross-reference, skim) costs 10-20 minutes for a decision that should take 30 seconds. Generic map apps return 200 pins with no curation; review sites bury good picks under ads and pagination.

## Users
**Primary user:** A casual trip planner or curious traveler — "person who just landed on a plan to visit Lisbon this weekend and wants a quick, trustworthy shortlist before diving into deeper research." Uses it on desktop or mobile, once per location, session-based (not a daily-use tool).

**Secondary users:** None implied by the request — this is a single-user, no-account utility.

**Anti-personas:**
- Professional travel agents needing exhaustive, bookable itineraries — wrong depth of tool.
- Users needing live data (current hours, real-time crowd levels, prices) — LLM recommendations are reasoned, not live-fetched.

## V1 scope
**In:**
- Text input for any location (free-form: city, region, landmark)
- Returns exactly 5 places with name + 1-2 sentence description
- Polished, responsive card-based UI (the visual bar is high per the request)
- Loading and error states (invalid/nonsense location, empty input)
- Fully stateless — no login, no database, no saved history

**Out (deliberate V1 omissions):**
- Real-time places API (Google Places/Foursquare) — no key management, no billing setup, keeps it zero-config — V2 if the user wants live data/photos/ratings
- Search history / saved favorites — adds a DB for no clear V1 payoff
- Maps/geolocation embed — nice-to-have, not requested
- Multi-language support — not requested

## Economics
- **Build:** under 1 engineer-day (single endpoint + static frontend, no infra to provision)
- **Fits budget:** YES — no third-party service costs; only cost is the hosting Claude API calls the app itself makes
- **Monthly infra:** hosting (~$0-5/mo on a free-tier host) + LLM inference cost per search (scales with traffic, no fixed fee)
- **Value:** collapses a 10-20 minute research task into a 5-second search

## Risks & dependencies
- **LLM hallucination risk** — Claude may occasionally invent a place name or misdescribe a real one — mitigate by prompting for only well-known, verifiable places and setting expectations in the UI copy ("AI-suggested — verify before you go").
- **No live data** — descriptions can go stale (a closed venue) — mitigated by keeping descriptions general/evergreen rather than time-sensitive (hours, prices).
- **Ambiguous location input** ("Springfield", "Paris" vs "Paris, TX") — handle by asking the LLM to pick the most likely/famous match and state its assumption in the response.

## DevOps & Deployment
- **Hosting target:** self-host / free-tier friendly (Fly.io, Render, or any VPS) — no vendor lock-in needed for a stateless single endpoint
- **Runtime + framework:** FastAPI + HTMX (Python), per the golden starter — server-rendered, minimal JS
- **Database & persistence:** none — fully stateless
- **Secrets & config:** `ANTHROPIC_API_KEY` (only needed if wired to live Claude API calls at runtime; for this build the seeded starter ships with a deterministic/mock recommendation function so it runs with zero keys, documented as the wire-up point)
- **Observability:** console logs only; add Sentry in Phase 2 if this gets real traffic
- **CI/CD:** manual deploy for V1 (single small app); GitHub Actions on push to main can be added later
- **Rollback story:** git revert + redeploy (no DB migrations to worry about)
- **Scaling envelope:** comfortably handles thousands of req/day on a single small instance; would need request queuing/caching well before that if LLM latency becomes a bottleneck

## Delivery
- **Phase 1 — Core experience** (Day 1): location input → 5-place results as styled cards, loading/error states. User can search any location and get a usable shortlist.
- **Phase 2 — Polish** (Day 1, same day): responsive layout, empty/edge-case handling, category-free clean card design, micro-interactions (hover, transition on results).

## Definition of done
- User can type any real-world location and receive exactly 5 named places with descriptions.
- Invalid/empty input shows a clear, non-broken error state.
- UI is responsive (mobile + desktop) and visually polished — not a bare form.
- App runs with zero required API keys/config out of the box (recommendation logic is self-contained or clearly mocked with a documented wire-up point).

## Open questions
None — the operator (this session's user) has already resolved the three load-bearing decisions: LLM-generated recommendations, name+description card format, stateless with no persistence.

## Assumptions made
- "Pretty" means a modern, card-based, polished visual design (not literally styling opinions beyond that) — confirmed by user intent, will realize via clean typography, spacing, and a cohesive color system.
- Recommendations should favor globally well-known, verifiable places over obscure ones, to minimize hallucination risk.
- Single language (English) input/output is sufficient for V1.
