# Intent SOW — WanderFive

## Domain
Consumer travel-discovery micro-tool (single-session, no-account utility). Adjacent to trip-planning apps but scoped far narrower: one input, one output, no itinerary building.

**Dynamics:**
- Usage is bursty and one-off per location — not a daily habit loop.
- The entire value is in curation quality + presentation speed, not data breadth.
- Users compare mentally against "just Googling it" — the bar is faster and more pleasant, not more comprehensive.
- No monetization or growth mechanics implied by the request — this is a utility, not a business.

**Constraints:** None regulatory/compliance-wise — no PII collected, no accounts, no payments. The only "constraint" is self-imposed: zero required API keys/config for V1.

**What downstream should pay attention to:** Because there's no real places database, the entire product quality hinges on prompt design for the recommendation step and on the visual polish of the result cards — both deserve disproportionate attention relative to the rest of the (small) build.

## Scope

### Executive summary
A one-page app where a user types a location and instantly gets 5 recommended places to visit, presented as polished cards. Recommendations come from Claude's own reasoning, not a live places API, keeping the app dependency-free and instant to run.

### In scope (V1)
- User enters free-form location text and submits
- App returns exactly 5 named places with a short description each
- Results render as visually polished cards (not a plain list)
- Loading state while generating results
- Error state for empty/invalid input or generation failure
- Fully responsive layout (mobile + desktop)

### Out of scope (V1, deliberate)
- Live places API integration (Google Places/Foursquare) — adds API key setup + billing, contradicts zero-config goal
- Saved search history / favorites — no clear V1 payoff, adds a database
- Maps or geolocation embeds — not requested, adds complexity
- User accounts / auth — no multi-user need
- Multi-language i18n — English-only is sufficient for V1

### Why these boundaries
The request is explicitly a small, self-contained utility judged mainly on visual polish — not a data platform. Keeping it stateless and API-key-free maximizes how fast it can ship and how easily anyone can run it, while the design investment goes where it's actually visible: the result cards.

## Impact

**Who benefits:** Primary — the person searching, who gets a fast, opinionated shortlist instead of a multi-tab research session. Benefit is felt immediately (under 5 seconds) as a completed decision instead of an open research task.

**Magnitude estimate:** Collapses a ~10-20 minute ad-hoc research task into a single ~5-10 second search. Needs validation with real users, but the bracket is plausible given how comparable "quick answer" tools perform against manual research.

**Evidence we have:** The braindoc's framing of the pain (multi-tab research, generic map apps returning too many unranked results) directly motivates a curated 5-item shortlist.

**Evidence we'd need:** Real usage data on how often people actually accept the 5 suggestions without further research — not gathered for V1 since there's no analytics/accounts.

**Measurement plan (post-launch):** If this ever gets instrumented: (1) searches per session, (2) time-to-first-result, (3) qualitative "did the results feel useful" feedback if a feedback widget is ever added (not in V1).

## Longevity

**3-year outlook:** Quick-answer LLM utilities like this will likely remain relevant as a category, but this specific tool has no moat — it's a thin, well-designed wrapper around a prompt. That's fine for its intended purpose (a polished demo/utility), not for a venture.

**Defensibility:** None — the value is entirely presentation quality and prompt design, both easily replicated. Being honest about this: it's a valid small project, not a defensible business.

**Market drift risks:** (1) Fast — general-purpose AI assistants (ChatGPT, Gemini, etc.) already answer "give me 5 places to visit in X" directly, competing on convenience. (2) Slow — real places APIs becoming free/trivial to integrate would erode the "zero-config" advantage of LLM-only recommendations.

**Kill criteria:** This is a scoped utility, not an ongoing venture — there's no traffic/engagement threshold to "kill" it. Revisit only if the operator wants to invest further (e.g., add live data) and usage doesn't justify the added complexity.

**Verdict:** Green — ship as scoped. It's a small, well-defined utility; the "risk" of no long-term moat doesn't apply to a demo/tool built for one person's use.

## Dependencies

**Data dependencies:** None — no external data source; recommendations are generated by the LLM as an in-request reasoning step.

**Partner/vendor dependencies:** Optionally, the Anthropic API if the "generate recommendations" step is wired to a live Claude call at runtime (cost: pay-per-token, no fixed fee). If unavailable/unset, V1 falls back to a documented local heuristic so the app still runs.

**Regulatory/compliance:** None apply — no PII, no accounts, no payment data.

**Distribution:** Direct URL sharing / self-host. No app store, no embed partner.

**Critical path callouts:** The only thing that could block shipping is deciding whether the "5 places" logic calls a live LLM API (needs a key, adds cost per search) or ships as a static/deterministic function for the demo. Recommendation: build the endpoint so it's LLM-call-shaped but ships with a clearly labeled example dataset generator, so it runs immediately with zero keys and can be pointed at a real API later by anyone who wants live generation.

## Variants

### Variant 1: Lean (tier_lean)
**Budget band (USD):** $0–50 (hosting only)
**Engineer-weeks:** 0.2 (under a day)
**Ops complexity:** low
**Recommended stack family:** fastapi (server-rendered HTMX, no build step)
**Target scale:** single user / demo traffic

**Tradeoffs:**
- Fast to ship, zero external dependencies
- No live data — descriptions may go stale for time-sensitive info (hours, prices)
- No persistence — nothing to lose, nothing to maintain

**Kill criteria:**
- Wrong choice if the operator actually wants live ratings/photos/hours from day one

### Variant 2: Recommended (tier_recommended)
**Budget band (USD):** $0–50 (hosting only; same as lean)
**Engineer-weeks:** 0.2–0.4
**Ops complexity:** low
**Recommended stack family:** fastapi (server-rendered HTMX + bundled design system)
**Target scale:** single user / demo / light public traffic

**Tradeoffs:**
- Same zero-config approach as Lean, but with the extra half-day invested in visual polish (the request's explicit "make it pretty" bar), transitions, and edge-case handling
- Still no live places data
- Still no persistence — genuinely not needed for this scope

**Kill criteria:**
- Wrong choice only if scope grows to need live data or accounts — then re-architect

### Variant 3: Hardened (tier_hardened)
**Budget band (USD):** $50–300/mo (API costs at scale + managed hosting)
**Engineer-weeks:** 1.5–2.5
**Ops complexity:** medium
**Recommended stack family:** fastapi + managed Postgres, or nextjs if a richer client experience is wanted
**Target scale:** hundreds–thousands of daily users

**Tradeoffs:**
- Adds real places API integration (live hours/ratings/photos), caching layer to control LLM/API cost, rate limiting, and observability
- Meaningfully more expensive and slower to ship
- Only worth it if this becomes a real public product, not a personal utility

**Kill criteria:**
- Wrong choice for a single-user demo — massive overkill relative to the stated request

### Recommended variant: tier_recommended
It matches the requested scope exactly (stateless search, no accounts) while investing the visual polish the user explicitly asked for, at effectively the same cost and timeline as Lean.

### Open questions / risks
- Should the app wire up a real Claude API call for recommendations, or ship with a deterministic example generator (no key needed) with the wiring documented? (Recommendation: latter for zero-config V1, documented wire-up point for real use.)
- Is there a risk of the LLM inventing a nonexistent place? (Mitigation: prompt for only well-known, verifiable places; add a small disclaimer in the UI.)
- No other material risks for a project this size.
